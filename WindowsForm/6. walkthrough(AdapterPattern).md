## 1. Introduction
This document explains how the **Adapter Pattern** is implemented within the data access layer of this project, specifically focusing on how the system manages the mismatch between the domain (`ICustomer`) and the object-relational mapper (**Entity Framework Core**).

## 2. What is the Adapter Pattern?
The Adapter Pattern acts as a bridge between two incompatible interfaces. In situations where two systems cannot communicate smoothly because their expected data forms or interfaces differ, the Adapter acts as a "translator" between the two.

**Classic Analogy:**
Imagine you are an English speaker traveling abroad.
- You speak English.
- A local restaurant only speaks French.
- The bilingual **Tour Guide** translates your English requests into French for the restaurant, and their French replies into English for you. **The Tour Guide is the Adapter.**

## 3. The Problem Context
In our system architecture:
- Our application logic (the **UI** and **MiddleLayer**) works strictly with the `InterfaceLayer.ICustomer` interface.
- Entity Framework (EF) Core, our database mapping framework, refuses to map directly to an interface (`ICustomer`). It absolutely requires a concrete class with specific entity annotations (in our case, `CustomerEntity`).

These two parts of our application cannot talk directly. Instead, we need something to translate back and forth between them.

## 4. The 4 Roles in the Adapter Pattern
The Adapter pattern works strictly with four distinct parts:

1. **Target (What our app wants to use):** 
   - This represents what your application expects to work with. In this system, the Target is the `ICustomer` interface (handled via the `IRepository<ICustomer>`).
   
2. **Adaptee (What the external system requires):** 
   - This represents the stubborn third-party requirement or foreign code that we cannot change. Here, it is **Entity Framework** mapping to a `CustomerEntity` format. `CustomerEntity` is the "language" EF understands.

3. **Adapter (The Translator):**
   - This is the **`EfRepository`** class. It sits right in the middle. It implements the Target application interface, but under the hood, it converts the objects back and forth to satisfy the Adaptee (`CustomerEntity`/EF).

4. **Client:** 
   - The UI or MiddleLayer that triggers data operations (e.g., calling `_repository.Add(customer)`). It has zero knowledge of `CustomerEntity` or EF.

## 5. The Adapter Pattern in Action

Let's look at exactly how `EfRepository` demonstrates the Adapter Pattern.

### Adding Data (Domain to Database)
When the Client wants to save an `ICustomer`, EF doesn't understand it. The `EfRepository` acts as the Adapter and converts the `ICustomer` to `CustomerEntity`:

```csharp
// EfRepository.cs -> ExecuteAdd
if (entity is InterfaceLayer.ICustomer customer)
{
    using (var context = new CustomerDbContext(ConnectionString))
    {
        // ADAPTER AT WORK: Translating from ICustomer object into CustomerEntity EF object!
        var entityToStore = new CustomerEntity
        {
            CustomerName = customer.CustomerName,
            PhoneNumber = customer.PhoneNumber,
            BillAmount = customer.BillAmount,
            BillDate = customer.BillDate,
            Address = customer.Address,
            // Saving the concrete type name out of the domain abstraction
            CustomerType = customer.GetType().Name 
        };
        context.Set<CustomerEntity>().Add(entityToStore);
        context.SaveChanges();
    }
}
```

### Reading Data (Database to Domain)
When reading, EF gives the application raw `CustomerEntity` rows back. Because the application strictly expects `ICustomer`, the `EfRepository` once again steps in to translate backwards:

```csharp
// EfRepository.cs -> ExecuteGetAll
var dbEntities = context.Set<CustomerEntity>().ToList();
var list = new List<T>();

foreach (var dbEntity in dbEntities)
{
    // ... null checking omitted for brevity ...
    
    // ADAPTER AT WORK: Translating the CustomerEntity back into an ICustomer object!
    var domainObj = CustomerFactory.Factory.Create(type);
    domainObj.CustomerName = dbEntity.CustomerName ?? "";
    domainObj.PhoneNumber = dbEntity.PhoneNumber ?? "";
    domainObj.BillAmount = dbEntity.BillAmount ?? 0;
    domainObj.BillDate = dbEntity.BillDate;
    domainObj.Address = dbEntity.Address ?? "";

    list.Add((T)domainObj);
}
```

## 6. Why The Others Are NOT the Adapter Pattern

It's common to confuse the Adapter pattern with other Structural or Behavioral patterns used heavily in the `Repository` package. Here is why `ExecuteAdd`, `ExecuteDelete` themselves, and `AdoRepository`, are fundamentally different:

### Template Method Pattern vs Adapter
- **What it is:** The `BaseRepository` defining `ExecuteAdd()`, `ExecuteDelete()`, etc., but deferring the implementation of those methods down to child classes. The base class controls the high-level workflow (the "Template").
- **Why it's not Adapter:** It defines *how* an algorithm is structured and extended across standard steps. It is not translating between two incompatible interfaces.

### Repository/Strategy Pattern vs Adapter
- **What it is:** Exposing a simple `IRepository` interface to the application, hiding whether EF, ADO, or Dapper is working behind the scenes.
- **Why it's not Adapter:** This acts as an interchangeable strategy so the rest of the application remains unchanged. The `AdoRepository` and `DapperRepository` both represent *Strategies*. As they interact directly with simpler SQL data formats (like DataReaders rows) without needing to negotiate two drastically different, incompatible object-oriented object trees directly, they do not illustrate the pure object-conversion concept of the Adapter pattern near as robustly as `EfRepository` mediating between `ICustomer` and EF's `CustomerEntity`.

---

## 7. Conclusion
In summary, **`CustomerEntity` is not the Adapter.** It is simply the required format for the Adaptee (EF). 

The true Adapter is **`EfRepository`**. It protects the core application by receiving standard `ICustomer` requests and invisibly translating them into `CustomerEntity` data just in time to satisfy Entity Framework Core's requirements.
