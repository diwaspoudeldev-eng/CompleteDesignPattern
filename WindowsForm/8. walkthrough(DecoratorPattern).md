## 1. Introduction
This document explains how we applied the **Decorator Design Pattern** to handle validation rules for different customer types. By using this pattern, we've moved away from rigid, monolithic validation classes to a flexible, composable system.

## 2. What is the Decorator Pattern?
The Decorator pattern allows behavior to be added to an individual object, dynamically, without affecting the behavior of other objects from the same class.

Think of it like **clothing**:
- You are the **Base Component**.
- A **Shirt** is a decorator that adds "warmth".
- A **Jacket** is another decorator that adds "waterproofing" on top of the shirt.
- You can mix and match these decorators without changing "who you are".

## 3. The Problem: Monstrous Validation Classes (Before)
Before applying the Decorator pattern, each customer type had its own hardcoded validation class.

**Example: `CustomerValidation`**
```csharp
public class CustomerValidation : IValidation<ICustomer>
{
    public void Validate(ICustomer entity)
    {
        if (string.IsNullOrWhiteSpace(entity.CustomerName)) throw new Exception("...");
        if (string.IsNullOrWhiteSpace(entity.PhoneNumber)) throw new Exception("...");
        if (entity.BillAmount == null || entity.BillAmount == 0) throw new Exception("...");
        if (entity.BillDate == null) throw new Exception("...");
        // This grows every time a new field is added!
    }
}
```

**Drawbacks:**
- **Code Duplication**: `LeadValidation` repeats the Name and Phone checks.
- **Rigidity**: If you want a new type with 80% of the same rules, you have to create a whole new class or use complex inheritance.
- **Violation of Open/Closed Principle**: To add a new validation rule, you have to modify the existing validation class.

## 4. The Solution: Composable Decorators (After)
We broke the validation logic into tiny, single-purpose **Decorators**.

- `NameValidationDecorator`
- `PhoneValidationDecorator`
- `AddressValidationDecorator`
- `BillValidationDecorator`

Now, we can "build" a validation chain just by wrapping these decorators together!

## 5. Validation Rules Table
The following table defines the requirements for each customer type:

| Type / Field | Customer Name | Phone Number | Bill Amount | Bill Date | Address |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Customer** | Compulsory | Compulsory | Compulsory | Compulsory | Compulsory |
| **Lead** | Compulsory | Compulsory | No Validation | No Validation | No Validation |
| **Self Service** | Compulsory | Compulsory* | No Validation | No Validation | No Validation |
| **Home Delivery**| Compulsory | No Validation | No Validation | No Validation | Compulsory |

*\*Self Service phone validation is assumed compulsory in this implementation.*

### How it maps to Decorators:
- **Customer**: `Name` + `Phone` + `Bill` + `Address` decorators.
- **Lead**: `Name` + `Phone` decorators.
- **Self Service**: `Name` + `Phone` decorators.
- **Home Delivery**: `Name` + `Address` decorators.

## 6. Implementation Details

### The Factory (The Assembly Line)
In `CustomerFactory.cs`, we now assemble these chains:

```csharp
case "Customer":
    cust = new Customer();
    cust.Validation = new AddressValidationDecorator(
                        new BillValidationDecorator(
                            new PhoneValidationDecorator(
                                new NameValidationDecorator(
                                    new BaseValidation<ICustomer>()))));
    break;

case "Home Delivery":
    cust = new HomeDelivery();
    cust.Validation = new AddressValidationDecorator(
                        new NameValidationDecorator(
                            new BaseValidation<ICustomer>()));
    break;
```

## 7. Conclusion
By using the Decorator pattern:
- **Maintenance is easier**: If the Name validation logic changes, we only edit `NameValidationDecorator`.
- **Flexibility**: Want a "VIP Customer" with extra validation? Just add one more decorator to the chain.
- **Clean Code**: Each class has exactly one reason to change (Single Responsibility Principle).
