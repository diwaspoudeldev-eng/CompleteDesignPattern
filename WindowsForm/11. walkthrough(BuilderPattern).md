# Walkthrough: Builder Design Pattern

We have implemented the **Builder Design Pattern** to provide a fluent and readable way to construct complex `ICustomer` objects.

## What is the Builder Pattern?
The Builder pattern separates the construction of a complex object from its representation, allowing the same construction process to create different representations (in our case, different Customer types).

## Implementation Details

### 1. The Interface: `ICustomerBuilder`
Defines the fluent methods for setting each property of a customer.
**File**: [ICustomerBuilder.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/InterfaceLayer/ICustomerBuilder.cs)

### 2. The Concrete Builder: `CustomerBuilder`
Implements the interface and uses the `Factory` to create the initial object instance before setting its properties.
**File**: [CustomerBuilder.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/MiddleLayer/CustomerBuilder.cs)

### 3. Fluent UI Integration
We refactored the `SetCustomer` method in the UI to use the new builder. Instead of setting properties line-by-line, we now "build" the object fluently.

**File**: [FormCustomer.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/WindowsForm/FormCustomer.cs)
```csharp
private void SetCustomer()
{
    cust = new CustomerBuilder(cmbCustomerType.Text)
        .WithId(...)
        .WithName(txtCustomerName.Text)
        .WithPhone(...)
        .WithBilling(...)
        .WithAddress(...)
        .Build(); // Final objective returned
}
```

## Benefits
1.  **Readability**: The sequence of method calls clearly describes what is being set.
2.  **Immutability Support**: Builders are excellent for creating immutable objects if needed in the future.
3.  **Step-by-Step Construction**: Unlike constructors with many parameters, you only set what you need, in any order.
4.  **Professionalism**: Before, the `SetCustomer` method was a list of direct assignments. Now, it reads like a sentence: *"Create a Customer of this type, with this Name, with this Phone, and with this Billing."* This makes the code much more professional and easier for other developers to follow.

---
> [!TIP]
> This pattern is often used alongside the **Factory Pattern**. Our `CustomerBuilder` uses the `Factory` to determine which concrete class to instantiate before populating it!


# Walkthrough: Report Builder Design Pattern

We have expanded the use of the **Builder Design Pattern** by implementing a reporting system that constructs a CSV document from the data in your Grid View.

## Why use Builder for Reporting?
Reporting is a perfect use case for the Builder pattern because reports are constructed in multiple steps:
1.  **Add Header** (column names)
2.  **Add Body Rows** (the actual data)
3.  **Add Footer** (totals or metadata)
4.  **Finalize** (get the completed string/document)

## Implementation Details

### 1. The Interface: `IReportBuilder`
Defines the steps required to build any report.
**File**: [IReportBuilder.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/InterfaceLayer/IReportBuilder.cs)

### 2. The Concrete Builder: `CsvReportBuilder`
Implements the steps to create a comma-separated format.
**File**: [CsvReportBuilder.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/MiddleLayer/CsvReportBuilder.cs)

### 3. UI Integration
A **"Generate CSV Report"** button was added. It iterates through the `DataGridView` and uses the builder to construct the report content.

**File**: [FormCustomer.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/WindowsForm/FormCustomer.cs)
```csharp
private void btnGenerateReport_Click(object sender, EventArgs e)
{
    IReportBuilder builder = new CsvReportBuilder();
    builder.AddHeader(new string[] { "ID", "Name", ... });
    
    foreach (var row in grid.Rows) {
        builder.AddRow(rowData);
    }
    
    builder.AddFooter("Total Records: " + count);
    string result = builder.GetReport();
    MessageBox.Show(result);
}
```

## Benefits of this Second Builder
While our first builder creates **objects** (`Customer`), this builder creates **documents** (`CSV String`). It shows the versatility of the pattern in handling different types of "complex constructions."

---
> [!TIP]
> You could easily add a `HtmlReportBuilder` or `PdfReportBuilder` that implements the same `IReportBuilder` interface, and the UI code wouldn't have to change!
