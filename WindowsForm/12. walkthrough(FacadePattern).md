# Walkthrough: Facade Design Pattern
The Facade Pattern is a structural design pattern that provides a simple interface to a complex system.
A facade is a wrapper class that hides complex logic behind a single, easy-to-use API.

👉 Instead of calling many classes directly, the client calls one facade.


We have implemented the **Facade Design Pattern** to simplify the interaction between the User Interface (UI) and the complex backend subsystems.

## The Problem
In the original `FormCustomer.cs`, the UI had to know about:
1. `CustomerBuilder` (to create objects)
2. `Validation` decorators (to validate data)
3. `Repositories` (to save to DB)
4. `CustomerCollection` (to manage in-memory items)
5. `ReportBuilder` (to generate CSV)

This resulted in a "Fat UI" with complex event handlers.

## The Solution: `CustomerFacade`
We created a **Facade** class that provides a simplified, higher-level interface to these subsystems.

**File**: [CustomerFacade.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/MiddleLayer/CustomerFacade.cs)

### Key Methods
- `AddDirect()`: One call that handles Building -> Validating -> Saving to DB.
- `AddToMemory()`: One call that handles Building -> Validating -> Adding to Collection.
- `BatchSave()`: Handles iterating over memory and saving to the chosen Repository.
- `GenerateCsvReport()`: Wraps the `ReportBuilder` complexity.

## Comparison: Direct vs Facade
To see the difference, compare these two files:

1.  **Direct Way**: [FormCustomer.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/WindowsForm/FormCustomer.cs)
    - Code is verbose.
    - UI is tightly coupled to many backend classes.
2.  **Facade Way**: [FacadeForm.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/WindowsForm/FacadeForm.cs)
    - Code is lean and clean.
    - UI only knows about the `CustomerFacade`.

### Visual Highlight: `btnAdd_Click`
**Before (Direct)**:
```csharp
private void btnAdd_Click(object sender, EventArgs e)
{
    SetCustomer(); // Manual build
    cust.Validate(); // Manual validate
    GetRepository().Add(cust); // Manual save
}
```

**After (Facade)**:
```csharp
private void btnAdd_Click(object sender, EventArgs e)
{
    _facade.AddDirect(type, id, name, ...); // One clean call
}
```

---

## Benefits
-   **Decoupling**: The UI doesn't break if you change a backend subsystem (e.g., swapping `XmlRepository` for `SqlRepository`).
-   **Simplified UI**: event handlers become 1-2 lines of code.
-   **Consistency**: Logic (like "Validate before Add") is centralized in the Facade, preventing bugs in multiple forms.
