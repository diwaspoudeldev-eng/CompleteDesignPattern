## 1. Introduction
This document explains the concept of the **Unit of Work (UOW)** pattern and explores whether or not it is necessary within the context of our current generic repository design, particularly when using Object-Relational Mappers (ORMs) like **Entity Framework Core**.

## 2. What is the Unit of Work (UOW) Pattern?
The Unit of Work pattern coordinates the writing out of changes and the resolution of concurrency problems. In essence, it maintains a list of objects affected by a business transaction and coordinates the writing out of changes, so that if one step fails, the entire transaction can be rolled back.

**Classic Analogy:**
Imagine you are at a grocery store.
- You place an apple, bread, and milk in your cart. You don't pay for each item individually as you pick it up.
- The **Checkout Counter** is the Unit of Work. You present all your items at once, and you pay for everything in one single transaction. If your card is declined, you leave with nothing (the transaction fails).

## 3. The Role of Unit of Work (UOW) in Data Access
When working with Data Access layers involving Repositories, the question often arises: **"Do we need a Unit of Work (UOW) pattern here?"**

**Important Note:** EF's `DbContext` **already implements UoW and Repository patterns** internally. You do not have to write your own custom UOW abstraction to get UOW behavior. UOW helps in committing all changes as a single transaction. When you call `SaveChanges()` on an EF `DbContext`, EF sends all those tracked changes to the database in one operation automatically.

## 4. In Our Current Repository Design
Right now, our repositories handle `SaveChanges()` directly. For example, in `EfRepository.cs`, when `ExecuteAdd` executes, it instantiates its own `CustomerDbContext` and immediately calls `context.SaveChanges()`.

This is fine for simple applications where a business transaction only involves a single table/entity, but as applications grow complex, this approach scales poorly. 

Let's look at the correct and incorrect ways to manage this in EF.

## 5. Common UOW Approaches with Entity Framework Core

### Approach 1: Safe - SaveChanges at the End
The simplest way to use UOW is to perform all your additions/modifications in memory and only call `SaveChanges` at the very end of the service method.

```csharp
public class OrderRepository
{
    private readonly AppDbContext _ctx;
    public OrderRepository(AppDbContext ctx) { _ctx = ctx; }

    public async Task AddAsync(Order order)
    {
        await _ctx.Orders.AddAsync(order); // Just tracks in memory
        // ✅ No SaveChanges here
    }
} 

public class InventoryRepository
{
    private readonly AppDbContext _ctx;
    public InventoryRepository(AppDbContext ctx) { _ctx = ctx; }

    public async Task DeductStockAsync(string product, int qty)
    {
        var item = await _ctx.Inventories.FirstAsync(i => i.Product == product);
        item.Stock -= qty; // Just modifies in memory
        // ✅ No SaveChanges here
    }
}

// Interacting Service
public class OrderService
{
    private readonly OrderRepository _orderRepo;
    private readonly InventoryRepository _inventoryRepo;
    private readonly AppDbContext _ctx;

    public OrderService(OrderRepository orderRepo, InventoryRepository inventoryRepo, AppDbContext ctx)
    {
        _orderRepo = orderRepo;
        _inventoryRepo = inventoryRepo;
        _ctx = ctx;
    }

    public async Task PlaceOrderAsync(string product, int qty)
    {
        await _orderRepo.AddAsync(new Order { Product = product, Quantity = qty });
        await _inventoryRepo.DeductStockAsync(product, qty);

        // ✅ ONE SaveChanges — ONE transaction!
        await _ctx.SaveChangesAsync(); 
    }
}
```

### Approach 2: Risky - SaveChanges Inside Repository
This is an **anti-pattern**. If you call `SaveChanges` inside every repository method immediately, you lose the ability to group them into a single transaction.

```csharp
public async Task AddAsync(Order order)
{
    await _ctx.Orders.AddAsync(order);
    await _ctx.SaveChangesAsync(); // ❌ Commits immediately!
}

public async Task DeductStockAsync(string product, int qty)
{
    var item = await _ctx.Inventories.FirstAsync(i => i.Product == product);
    item.Stock -= qty;
    await _ctx.SaveChangesAsync(); // ❌ Commits immediately!
}

public async Task PlaceOrderAsync(string product, int qty)
{
    // If AddAsync succeeds, but DeductStockAsync throws an exception...
    await _orderRepo.AddAsync(new Order { Product = product, Quantity = qty });
    
    // ...the stock is never deducted, but the order is permanently saved! Data is inconsistent.
    await _inventoryRepo.DeductStockAsync(product, qty);
}
```

### Approach 3: The Danger of Different DbContexts
Even if you wait to call `SaveChanges` at the end, it breaks if every repository news up its own `DbContext` instance.

```csharp
// ❌ THE PROBLEM — Each repo creates its OWN DbContext
public class OrderRepository
{
    private readonly AppDbContext _ctx = new AppDbContext(); // Own instance!
    public async Task AddAsync(Order order) { await _ctx.Orders.AddAsync(order); } // Tracked in ctx_A
}

public class InventoryRepository
{
    private readonly AppDbContext _ctx = new AppDbContext(); // Different instance!
    public async Task DeductStockAsync(...) { ... } // Tracked in ctx_B
}

public class OrderService
{
    private readonly AppDbContext _ctx = new AppDbContext(); // Yet another instance!

    public async Task PlaceOrderAsync(string product, int qty)
    {
        await _orderRepo.AddAsync(...);        // Changes in ctx_A
        await _inventoryRepo.DeductStockAsync(...); // Changes in ctx_B

        await _ctx.SaveChangesAsync(); // ctx_C — knows about NOTHING!
        // ❌ Saves NOTHING because this context has no tracked changes
    }
}
```

### Approach 4: Safe - Shared DbContext via Scoped DI
The easiest way to fix Approach 3 is to rely on **Dependency Injection (DI)** with a `Scoped` lifetime.

In .NET Core / .NET 5+, registering a service as `Scoped` ensures that only **ONE instance** of that class is created per HTTP Request.

```csharp
// DI Registration in Startup.cs / Program.cs
services.AddScoped<AppDbContext>();         // ← KEY: Scoped! ONE instance per request
services.AddScoped<OrderRepository>();
services.AddScoped<InventoryRepository>();
services.AddScoped<OrderService>();
```
Because it is Scoped, the DI container injects the exact same `AppDbContext` instance into `OrderRepository`, `InventoryRepository`, and `OrderService`. When `OrderService` finally calls `_ctx.SaveChangesAsync()`, it encompasses all changes made across all repositories.

### Approach 5: Safe - Explicit Unit of Work Interface
If you want to abstract away knowledge of `DbContext` entirely from your Service layer, you can create a literal `IUnitOfWork` interface. The UOW is responsible for creating/managing the `DbContext` and distributing it to the repositories.

```csharp
// UoW Interface
public interface IUnitOfWork : IDisposable
{
    IOrderRepository Orders { get; }
    IInventoryRepository Inventory { get; }
    Task CommitAsync();
}

// UoW Implementation — creates ONE DbContext, passes to ALL repos
public class UnitOfWork : IUnitOfWork
{
    private readonly AppDbContext _ctx;
    public IOrderRepository Orders { get; }
    public IInventoryRepository Inventory { get; }

    public UnitOfWork(AppDbContext ctx)
    {
        _ctx = ctx;                               // ONE context
        Orders    = new OrderRepository(_ctx);    // Passed to OrderRepo
        Inventory = new InventoryRepository(_ctx); // Same context passed
    }

    public async Task CommitAsync()
    {
        await _ctx.SaveChangesAsync(); // ONE commit for everything ✅
    }
    public void Dispose() => _ctx.Dispose();
}

// Service — only depends on IUnitOfWork, NO DbContext knowledge
public class OrderService
{
    private readonly IUnitOfWork _uow;

    public OrderService(IUnitOfWork uow) { _uow = uow; }

    public async Task PlaceOrderAsync(string product, int qty)
    {
        try
        {
            await _uow.Orders.AddAsync(new Order { Product = product, Quantity = qty });
            await _uow.Inventory.DeductStockAsync(product, qty);

            await _uow.CommitAsync(); // ONE transaction — all or nothing ✅
        }
        catch (Exception) { /* Handle failure, nothing was saved */ }
    }
}

// DI Registration
services.AddScoped<AppDbContext>();
services.AddScoped<IUnitOfWork, UnitOfWork>();
services.AddScoped<OrderService>();
// Note: Repos are NOT registered separately — UoW creates them internally
```

## 6. Final Comparison

Here is a quick reference guide to evaluating the safety of these patterns:

| Example | Safe? | Why |
| :--- | :---: | :--- |
| **1. Add only, SaveChanges at end** | ✅ Safe | Tracks everything in one context, writes in one transaction. |
| **2. SaveChanges inside each repo** | ❌ Risky | Results in partial saves if an error occurs mid-operation; no rollback. |
| **3. Different ctx → fixed by Scope** | ✅ Safe | DI ensures a shared context is injected everywhere automatically. |
| **4. Different ctx → fixed by UoW** | ✅ Safe | The UoW explicitly owns and shares the context to its child repositories. |

## 7. Transactions Across Multiple Databases

The approaches above (`DbContext` natively functioning as a UOW) work beautifully when communicating with a **single database.** 

However, if your business transaction spans **two or more completely different databases** (e.g., saving data to `OrdersDB` and auditing logs to `LogsDB`, managed by two different DbContexts), a single `SaveChanges()` won't cover both simultaneously. If the first succeeds and the second fails, you are out of sync.

In these advanced scenarios, you cannot rely simply on the ORM's Unit of Work. You must wrap the entire operation in a `System.Transactions.TransactionScope` to enlist multiple database connections into a distributed transaction.

---

## 8. Conclusion
For 95% of modern .NET applications using Entity Framework Core, injecting a **Scoped Component** of `DbContext` (Approach 4) provides all the data consistency and Unit of Work transactional behaviors you need without adding the boilerplate of a custom `IUnitOfWork` wrapper (Approach 5). Only introduce the explicit wrapper if strict architectural decoupling is mandated.
