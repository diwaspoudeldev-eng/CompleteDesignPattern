## 1. Single Responsibility Principle (SRP)
> *A class should have only one reason to change.*

### Handler: Repository Pattern
We moved database logic out of the UI and into specialized repositories.

**Before SRP (The "Fat UI" approach):**
The Form would handle both UI logic and database queries. If the query changed, you had to recompile the UI.
```csharp
// Inside FormCustomer.cs
private void btnAdd_Click(...) {
    // UI logic
    if(string.IsNullOrEmpty(txtName.Text)) return;
    
    // Database logic (THIS VIOLATES SRP)
    using(var conn = new SqlConnection(...)) {
        var cmd = new SqlCommand("INSERT INTO Customers...", conn);
        cmd.ExecuteNonQuery();
    }
}
```

**After SRP (Our Current Code):**
The UI only handles the button click, and the **Repository** handles the SQL.
- **UI**: [FormCustomer.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/WindowsForm/FormCustomer.cs) calls `_repo.Add(cust)`.
- **Logic**: [AdoRepository.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/Repository/AdoRepository.cs) contains the SQL mapping.

## 2. Open/Closed Principle (OCP)
> *Software entities should be open for extension, but closed for modification.*

### Handler: Factory Pattern
We can add new customer types by creating new classes, without changing the existing logic.

**Before OCP:**
Every time you added a new customer type, you had to find every `if/else` block in your system and add a new case.

**After OCP (Our Current Code):**
We use the [CustomerFactory.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/CustomerFactory/CustomerFactory.cs). To add a "VIP" customer, you just create a `VIP` class and add one line to the Factory. The rest of the app "just works".
```csharp
public static ICustomer Create(string typeOfCustomer) {
    switch (typeOfCustomer) {
        case "Lead": return new Lead();
        case "Customer": return new Customer();
        case "SelfService": return new SelfService();
        // Just add new types here ^. Existing code is CLOSED to modification.
    }
}
```

## 3. Liskov Substitution Principle (LSP)
> *Objects of a superclass should be replaceable with objects of its subclasses.*

If code works with an interface or base class <- Superclass , it should work correctly with any of its implementations <--baseclass — without knowing which one it is.


### Handler: RIP (Replace If with Polymorphism)
We ensure that any object implementing `ICustomer` can be used wherever an `ICustomer` is expected.

**Violation of LSP:**
If `Lead` didn't implement `Validate()` correctly or threw an error that `Customer` didn't, the code would break when switching types.

**LSP compliance in our code:**
We use polymorphism to treat all subtypes identically. Whether it is a `Lead` or a `HomeDelivery` customer, the UI handles them the same way.
```csharp
// The UI doesn't care about the concrete type.
// It just calls the common interface methods.
ICustomer cust = Factory.Create(cmbType.Text);
cust.Validate(); // Every subtype is guaranteed to handle this call!
```

## 4. Interface Segregation Principle (ISP)
> *No client should be forced to depend on methods it does not use.*

### Handler: Multiple Interfaces (ReportBuilder)
We split big interfaces into smaller, meaningful ones like `ICustomer`, `IValidation`, and `IReportBuilder`.

**Violation of ISP:**
A single `ICustomer` interface that forced you to implement `GenerateCSVReport()` methods. Even if you just wanted to display a name, you'd have to write report logic.

**ISP compliance in our code:**
We created a separate [IReportBuilder.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/InterfaceLayer/IReportBuilder.cs) for reporting tasks. 
```csharp
public interface IReportBuilder {
    void AddHeader(string[] headers);
    void AddRow(string[] columns);
    void AddFooter(string footerText);
    string GetReport();
}
```
The `Customer` class stays clean because it doesn't need to implement these reporting methods.

## 5. Dependency Inversion Principle (DIP)
> *Depend upon abstractions, not concretions.*

### Handler: DIP & IOC (Inversion of Control)
The UI doesn't "choose" its database; it is "injected" with one from the outside.

**Scenario in our code:**
The [Program.cs](file:///d:/Projects/Personal/Practice/CompleteDesignPattern/WindowsForm/Program.cs) (the "Composition Root") creates the concrete repository and gives it to the Form.

```csharp
// Program.cs
IRepository<ICustomer> adoRepo = new AdoRepository<ICustomer>(connStr);
Application.Run(new FormCustomer(adoRepo, efRepo)); // Injection!

// FormCustomer.cs
public FormCustomer(IRepository<ICustomer> repo) { ... } // Depending on Abstraction!
```
The Form depends on `IRepository` (Interface), not `AdoRepository` (Concrete). We can swap ADO for Entity Framework by changing just one line in `Program.cs`.
